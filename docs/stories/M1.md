# M1 — Champion reviews and completes their team profile

**Status:** Not started
**Size:** L
**Enables:** M2 (Program PM dashboard reads completion signal across the fleet)

## User story

> As a **Champion** for a team migrating from Azure DevOps to GitHub,
> I open my team's profile, see what a discovery agent has auto-populated, flagged as needing input, and noted as anomalies,
> I answer the review questions for my team,
> and I see my team's profile move toward "human-verified" with a visible completion signal.

## Why this story is first

M1 is the demo spine. It exercises every section of the Team model end-to-end — identity, codebase, workflows, JTBDs, customizations, risks, ownership — across all 6 seeded cohorts. If M1 renders cleanly for every cohort, the schema is proven as the spine; every later story becomes a variation on the same surface. See [docs/DONE.md](../DONE.md) for the prototype-done flow that M1 plus seed data satisfies.

## In scope

1. **Team List page** at `/teams` rendering 6 teams (one per cohort) with name, cohort badge, migration state, and completion %.
2. **Team Profile page** at `/teams/[slug]` rendering all seven profile sections (identity, codebase, workflows, JTBDs, customizations, risks, ownership) from the DB for the loaded team.
3. **Agent Findings view** (tab or panel on the Team Profile) grouping the team's latest `AgentRun` findings into three buckets:
   - auto-populated (`FindingSource = ADO_API | INFERRED`, `FindingStatus = PENDING`)
   - needs-input (`FindingSource = NEEDS_INPUT`)
   - anomalies (`Confidence = LOW`)
4. **Review flow** from the Team Profile, presenting exactly three questions for the loaded team:
   - one single-select (example: primary build pipeline type)
   - one multi-select (example: which JTBDs the team performs)
   - one free-text (example: describe a customization the agent flagged)

   The review writes the human's answer back to the relevant `Finding` rows, transitioning `FindingStatus` from `PENDING` to `ACCEPTED` / `CORRECTED` / `OVERRIDDEN` as appropriate.

5. **Completion % and "human-verified" markers** on the Team Profile, derived from `Finding.status` transitions. Completion is a pure function of schema state — not a denormalized column.
6. **Seed data** for all 6 cohorts (Alpha, Bravo, Charlie, Delta, Echo, Foxtrot) with variance designed to stress-test every UI state on every page above. Per [docs/DONE.md](../DONE.md) — "variance IS the story."

## Out of scope (do not build in M1)

- Real authentication — a stubbed current user is fine. Auth.js wiring is M2.
- Real ADO API integration — seed rows in `AgentRun` and `Finding` stand in for the agent. Live discovery is M3.
- Ada chat, cut-over runbook, cohort/wave admin, post-migration validation, fleet risk view — M4 through M8.
- Email / notifications.
- Mobile polish beyond "doesn't break catastrophically."

## Acceptance criteria

A reviewer, given only the deployed URL, completes this flow in under 5 minutes without help:

1. Lands on `/teams` and sees 6 teams, one per cohort, with distinct completion percentages.
2. Clicks a team (Charlie or Delta is the interesting case — pick a cohort with a mix of auto-populated, needs-input, and anomaly findings) and lands on `/teams/[slug]` with all seven profile sections rendered from the DB.
3. Clicks "Agent Findings" and sees three grouped lists (auto-populated / needs-input / anomalies) with counts matching the seeded `AgentRun` summary counts.
4. Clicks "Review," answers three questions (single-select, multi-select, free-text), and submits.
5. Returns to the Team Profile and sees completion % increase and the reviewed fields marked "human-verified."

Every cohort renders every page above without breaking — including cohorts with sparse sections (no customizations, no risks, archived-only repos, etc.). Empty states are designed, not accidental.

## Technical notes

- **Reads:** Server Components only. No client-side fetch for profile data.
- **Writes:** the Review mutation is the first write in the app. Introduce tRPC here; do not add tRPC before it is needed.
- **Completion %:** pure function over `Team` + relations. No denormalized column on `Team`. If M2's dashboard needs fast reads later, add a read-model then — not now.
- **"Human-verified":** a `Finding` is human-verified when `updatedSource = HUMAN`. `FindingStatus != PENDING` alone is not sufficient — an agent could in principle move a finding out of `PENDING`, and the UI needs to distinguish human review from agent action. The `updatedBy` / `updatedSource` columns are added to `Finding` as part of M1's scaffolding PR, fixing schema-health smell 09 for this model.
- **Validation:** every tRPC input parsed with Zod. Zod schemas colocated with the router.
- **Styles:** Tailwind utility classes only. No new CSS files.
- **Components:** before adding a new component, search `src/components/` for an existing one that fits (CLAUDE.md hard rule).

## Prerequisites (must land before M1 claims done)

1. `prisma/seed.ts` exists and `npm run db:seed` succeeds against a fresh DB.
2. `tsx` added to `package.json` devDependencies — `db:seed` currently references it but it is not installed.
3. [docs/testing.md](../testing.md) exists with the unit / component / E2E split and conventions.
4. `Finding.updatedBy` and `Finding.updatedSource` exist in `prisma/schema.prisma` with an applied migration (closes schema-health smell 09 for this model).

These are setup, not feature work. They can land in a single scaffolding PR before M1's feature PRs start.

## Cohort variance (seed design)

Each cohort stress-tests a different slice of the UI. Exact shapes to be pinned in the seed PR, but the intent is:

| Cohort  | Shape                                                             |
| ------- | ----------------------------------------------------------------- |
| Alpha   | Small, clean: one repo, standard yaml pipeline, no customizations |
| Bravo   | Medium, healthy: multiple repos, yaml pipelines, low-risk         |
| Charlie | Messy: classic pipelines, custom tasks, multiple open risks       |
| Delta   | Large: LFS, submodules, long-running release pipelines            |
| Echo    | Partial: several needs-input findings, one high-risk anomaly      |
| Foxtrot | Edge: archived repos, orphaned ownership, rolled-back state       |

Variance is load-bearing. Six near-identical cohorts prove nothing.

## Resolved product decisions

These were open questions at story draft; resolved during kickoff.

- **Cohort shapes** — the six vignettes in the table above are the approved demo lineup. Charlie is the reviewer's first-pick team (richest mix of auto-populated / needs-input / anomalies).
- **Review questions** — **team-specific**, drawn from each team's own needs-input findings. Yields a richer demo, especially on Charlie (4 needs-input findings) and Echo (6).
- **Agent Findings layout** — **tabs**, not inline panel. The Team Profile has three tab routes: `/teams/[slug]` (Overview), `/teams/[slug]/findings` (Agent Findings), `/teams/[slug]/review` (Review). Each is a distinct screen so the Champion's journey reads as three deliberate moves, and the demo walkthrough has clear landmarks to point at.
