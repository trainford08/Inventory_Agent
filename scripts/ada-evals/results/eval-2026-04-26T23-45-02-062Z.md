### ✅ framework-p3-vs-p4 _(framework)_

**Q:** What's the difference between P3 and P4?
**Tools:** _none_ · **15335ms**

**Answer:**

```
**P3 (Compositional)** and **P4 (Decompositional)** are opposites in how they map data between ADO and GitHub:

- **P3 Compositional**: Multiple GitHub fields combine to reconstruct a single ADO capability. You're composing the ADO concept from several GitHub pieces. Example: ADO's rich work item hierarchy might map to a combination of GitHub Issues + labels + project fields.

- **P4 Decompositional**: One ADO field carries multiple meanings that must be split across different GitHub fields. You're decomposing the ADO field into separate GitHub concepts. Example: an ADO "State" field that encodes both workflow stage *and* resolution reason might split into GitHub status + a custom label.

Think of it as **many→one (P3)** vs. **one→many (P4)** in terms of the mapping direction from GitHub back to ADO.
```

### ✅ framework-ab-syntax _(framework)_

**Q:** What's AB#123 and why do I need it?
**Tools:** _none_ · **6545ms**

**Answer:**

```
**AB#123** is the linking syntax that connects GitHub pull requests to Azure DevOps work items in the hybrid model. When you include `AB#123` in a commit message or PR description, GitHub automatically creates a link back to ADO work item 123.

You need it because **Boards stays in ADO** while **Repos moves to GitHub**. AB# syntax preserves traceability between your code changes and your work items across the two platforms. Without it, you lose the ability to see which PRs delivered which stories, bugs, or tasks—critical for compliance, release notes, and velocity tracking.

This is one of the most common hybrid friction points, since developers must remember to include the syntax and teams often need to update templates, branch policies, or CI checks to enforce it.
```

### ✅ framework-gap-meaning _(framework)_

**Q:** What does 'gap' mean for capability preservation?
**Tools:** _none_ · **5651ms**

**Answer:**

```
**Gap** means the ADO capability cannot be preserved in GitHub at all—there's no equivalent feature, workaround, or compositional substitute that delivers the same outcome.

In the migration pattern taxonomy, this is **P6 (Gap)**. When an entity or field is marked with gap-level data preservation, teams lose that functionality entirely and must either change their workflow, accept the loss, or keep that piece in ADO under the hybrid model.

Gap-parity customizations and gap-preservation entities are the highest-friction items in the migration because they force hard trade-offs rather than technical mapping exercises.
```
