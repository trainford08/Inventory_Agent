### ✅ framework-p3-vs-p4 _(framework)_

**Q:** What's the difference between P3 and P4?
**Tools:** _none_ · **7281ms**

**Answer:**

```
**P3 (Compositional)** means **multiple GitHub fields combine** to reconstruct a single ADO capability. You're building *up* from smaller pieces to match what one ADO feature did.

**P4 (Decompositional)** is the reverse: **one ADO field carries multiple meanings** that must be split apart into separate GitHub fields. You're breaking *down* an overloaded concept.

**Example**: If ADO's "Reason" field holds both close-reason *and* state-transition logic, and GitHub uses separate `state_reason` and workflow triggers, that's **P4**—decomposing one field into two. If GitHub requires you to set `state`, `state_reason`, *and* a label to replicate ADO's "Resolved" status, that's **P3**—composing three things to match one.
```

### ✅ framework-ab-syntax _(framework)_

**Q:** What's AB#123 and why do I need it?
**Tools:** _none_ · **6443ms**

**Answer:**

```
**AB#123** is the syntax GitHub uses to link pull requests back to Azure DevOps work items in a hybrid migration scenario—where **Repos** and **Pipelines** move to GitHub, but **Boards** (and often Test Plans) stay in ADO.

When you include `AB#123` in a commit message or PR description, GitHub's [Azure Boards integration](https://github.com/marketplace/azure-boards) creates a link from that PR to ADO work item #123, preserving traceability across the two systems. Without it, your code changes and work tracking become disconnected.

This matters most for teams whose **Boards** JTBDs (planning, tracking, reporting) stay in ADO while development moves to GitHub—essentially every team following the standard hybrid model.
```
