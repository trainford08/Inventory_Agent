### ✅ framework-p3-vs-p4 _(framework)_

**Q:** What's the difference between P3 and P4?
**Tools:** _none_ · **12339ms**

**Answer:**

```
**P3 (Compositional)** means multiple GitHub fields or features must be combined to reconstruct a single ADO capability. You're building up the equivalent from smaller pieces. For example, ADO's rich work-item linking might map to a combination of GitHub Issues references + Projects + custom fields.

**P4 (Decompositional)** is the reverse: one ADO field carries multiple meanings or purposes, and those must be split across different GitHub fields. You're unpacking overloaded data. For instance, an ADO "Reason" field that tracks both state-change rationale and closure category might decompose into separate GitHub fields or labels.

In short: **P3 = many → one** (combine GitHub pieces to match ADO), **P4 = one → many** (split ADO's overloaded field into GitHub's discrete fields).
```

**Tools:** _none_ · **5561ms**

**Answer:**

```
**AB#123** is the linking syntax that connects GitHub pull requests to Azure DevOps work items in the hybrid model.

When you include `AB#123` in a PR title or commit message (where 123 is the ADO work item ID), GitHub automatically creates a link back to that work item in Azure Boards. This is critical because **Repos and Pipelines migrate to GitHub, but Boards stays in ADO**—so engineers need a way to trace code changes back to the user stories, tasks, or bugs that drove them.

Without it, you lose traceability between your code and your backlog, breaking audit trails and making it harder to understand *why* a change was made.
```
