### ✅ framework-p3-vs-p4 _(framework)_

**Q:** What's the difference between P3 and P4?
**Tools:** _none_ · **24582ms**

**Answer:**

```
**P3 (Compositional)** means that **multiple GitHub fields or features** must be combined together to reconstruct a single ADO capability. You're building up the ADO concept from several smaller GitHub pieces.

**P4 (Decompositional)** is the reverse: **one ADO field** carries multiple meanings or purposes, and those get split apart into separate GitHub fields. You're breaking down the ADO concept into distinct GitHub pieces.

**Example:**
- P3: ADO's rich work item linking might require GitHub issues + project boards + labels working together.
- P4: ADO's "State" field might encode both workflow stage *and* resolution reason, which GitHub splits into separate status and close reason fields.
```

### ✅ framework-ab-syntax _(framework)_

**Q:** What's AB#123 and why do I need it?
**Tools:** _none_ · **5574ms**

**Answer:**

```
**AB#123** is the syntax used to link GitHub pull requests to Azure DevOps work items in the hybrid model—where Repos and Pipelines move to GitHub but Boards stays in ADO.

When you include `AB#123` in a commit message or PR description (where 123 is the ADO work item ID), Azure Boards automatically creates a link back to that commit or PR. This preserves traceability between code changes in GitHub and work tracking in ADO, which is critical for teams doing **Pull Request linking (J16)** and relying on **Work item integration (F19)**.

Without it, engineers lose the automatic connection between their PRs and the user stories or bugs they're resolving, breaking audit trails and deployment tracking workflows.
```

### ✅ framework-gap-meaning _(framework)_

**Q:** What does 'gap' mean for capability preservation?
**Tools:** _none_ · **5172ms**

**Answer:**

```
**Gap** means the ADO capability cannot be preserved in GitHub — there's no equivalent feature or workaround that achieves the same outcome.

In the migration patterns, this is **P6 Gap**. When a feature or entity is marked as gap, teams must either abandon that workflow, keep using ADO for that piece (hybrid model), or build a custom solution outside the platform.

For example, if ADO offers a specialized approval gate that GitHub Actions simply doesn't support, and no combination of GitHub features can substitute it, that's a gap.
```
