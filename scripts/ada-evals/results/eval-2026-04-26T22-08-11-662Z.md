### ✅ framework-p3-vs-p4 _(framework)_

**Q:** What's the difference between P3 and P4?
**Tools:** _none_ · **5857ms**

**Answer:**

```
**P3 (Compositional)** means you combine **multiple GitHub fields** to reconstruct a single ADO capability. You're building up the ADO behavior from several GitHub pieces.

**P4 (Decompositional)** is the opposite: **one ADO field** carries multiple meanings, and you split it across several GitHub fields. You're breaking down an overloaded ADO concept into cleaner, separate parts.

**Example:** P3 might be using labels + milestones + projects together to replicate ADO's iteration path. P4 might be splitting ADO's "Assigned To" (which can mean both owner *and* reviewer) into separate GitHub assignee and reviewer fields.
```

### ✅ framework-ab-syntax _(framework)_

**Q:** What's AB#123 and why do I need it?
**Tools:** _none_ · **6881ms**

**Answer:**

```
**AB#123** is the syntax used to link GitHub pull requests to Azure Boards work items in a hybrid migration scenario.

When your team moves code (Repos) and CI/CD (Pipelines) to GitHub but keeps work-item tracking in Azure Boards, you lose the automatic traceability that existed when everything lived in ADO. By including `AB#123` (where 123 is a work-item ID) in your commit message or PR description, the Azure Boards + GitHub integration creates a clickable link from the PR back to the work item—and from the work item to the PR.

This matters because many JTBDs depend on traceability: engineers need to see what code changed for a bug fix, managers need to track which work items shipped in a release, and auditors need proof that features were tested. Without AB# syntax, that context is lost and teams resort to manual lookups or spreadsheets.
```

### ✅ framework-gap-meaning _(framework)_

**Q:** What does 'gap' mean for capability preservation?
**Tools:** _none_ · **5764ms**

**Answer:**

```
**Gap** means the Azure DevOps capability cannot be preserved in GitHub—there is no equivalent feature or workaround that delivers the same outcome.

In the migration patterns, this is **P6 (Gap)**. When a feature maps to P6, teams must either abandon that workflow, find a third-party solution, or keep that piece of work in Azure DevOps under the hybrid model.

Gap capabilities create the hardest migration decisions because they force process change, not just tooling translation.
```
