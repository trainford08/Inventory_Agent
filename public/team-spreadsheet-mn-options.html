M:N handling — three options

Three ways to handle the same data: a feature shared by multiple JTBDs (F05 used by J12 and J16) and an entity shared by multiple features (E16 used by F01 and F02). Same shape, different visualizations.

Scenario: Charlie's profile. F05 · Pull request review is used by both J12 · Open a pull request and J16 · Link a PR to a work item. E16 · Branch sits under both F01 · Git commit & push and F02 · Branch management. The strict tree breaks; we have to make a choice.
Option 2
Reference rows in the spreadsheet
Item appears fully under one parent, then ghosted under every other parent it serves with an arrow back to the authoritative row. The tree stays intact; reuse is honest but quiet.
+ Tree shape preserved; click-and-scan works.   + Edits happen in one place.   − M:N is implied, not visible at a glance.

Option 4
Matrix / coverage view
Rows = features, columns = JTBDs, cells = used (●) / shared (○) / orphan (!). M:N is the point. Use to spot orphans (features with no JTBD), hot features (used by many), and JTBDs with thin coverage.
+ M:N is fully visible.   + Surfaces orphans and coverage gaps that the tree hides.   − Useless for browsing detail; tells coverage, not story.
