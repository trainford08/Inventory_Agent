What changes in the seed

Each team gets a customizations array in the seed where team-specific entries carry the same fields as the catalog. Schema-wise, nothing changes — the catalog is just extended at seed time per team, or the team's customization rows carry the metadata directly.

// prisma/cohorts/charlie.ts (sketch)
customizations: [
  { catalogCode: "C01" },                      // inherits all framework fields
  { catalogCode: "C09" },
  {
    name: "Charlie cost-center field on bugs",
    scope: "TEAM_SPECIFIC",
    parity: "PARTIAL",
    strategy: "S04_REBUILD_WITH_LOSS",
    jobsToBeDone: "Tags bugs for finance chargeback reporting.",
    githubEquivalent: "Issue Field 'cost-center' (org-level)",
    notes: "Used for chargeback reporting; only Charlie...",
    metaCandidates: [
      { code: "S06", trigger: "If finance asks every team..." }
    ]
  },
]

Tradeoffs

Upside: No empty cells. Every row tells a complete story. Team-specific entries sit alongside catalog rows with the same level of analysis. Discovery output becomes a structured artifact, not a free-text list.

Downside: Discovery becomes a structured exercise (each team scores its own customizations against the same fields). For prototype data, we'd seed a small set per cohort by hand.

Recommendation: If you adopt this, lean into the Team-specific scope tag so users can still filter "framework-only vs everything." The orange tinting in the mock above is one option for that visual cue.
